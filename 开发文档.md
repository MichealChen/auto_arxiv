# auto_arxiv 开发文档

## 1. 项目定位

`auto_arxiv` 是一个面向个人研究者的 arXiv 每日文献推荐应用。系统每天自动获取指定研究领域的新论文，根据用户兴趣配置进行筛选、排序和摘要整理，并在程序页面中展示每日推荐结果。

Markdown 日报只是当前早期阶段的临时过渡产物，用于验证抓取、推荐、排序和内容组织链路。项目最终目标不是停留在静态 `.md` 报告，而是形成一个可本地运行、可 GitHub 部署、也可以封装成桌面应用的程序化产品。

## 2. 产品形态

项目需要同时考虑三种运行形态：

### 2.1 本地 Web 应用

- 用户在本机启动服务。
- 浏览器打开本地页面查看每日推荐。
- 推荐数据保存在本地文件或轻量数据库中。
- 适合个人长期使用和开发调试。

### 2.2 GitHub 部署

- 项目可以通过 GitHub Actions 定时抓取和生成推荐数据。
- 项目可以通过 GitHub Pages 或类似方式展示静态页面。
- GitHub 形态应适合无服务器、低成本部署。
- 用户可以通过仓库 secret 配置私有兴趣、API Key 或推送配置。

### 2.3 桌面应用封装

- 当前采用 Python 桌面应用 + 核心推荐逻辑复用方案。
- exe 启动后应在窗口内提供账户/Profile 管理、配置编辑、日历日期选择、推荐列表、论文详情、历史记录、待读列表、笔记、PDF 下载和退出能力。
- 后续可选升级方案包括 Electron 或 Tauri。
- 桌面端应支持一键刷新、查看历史、下载论文和导出推荐结果。
- 桌面端不应依赖浏览器作为主要交互入口。
- 桌面端的论文详情应采用结构化阅读面板，而不是纯文本堆叠。
- 桌面端应支持多个账户/Profile 档案切换，不要求真实联网登录。
- 桌面端日期选择应由输入框直接触发日历，日历只允许选择不晚于当天的日期，并出现在输入框附近；选择日期或关闭弹窗后应正确退出日历。
- 某日推荐不应只查询精确当天提交的论文，应使用截至该日期的滚动窗口，避免周末、假期或 arXiv 索引延迟导致“抓取 0、推荐 0”。
- 桌面端应支持按时间范围批量生成推荐；范围内每天展示 `limit` 条常规推荐，最终列表规模按天数累计。
- 桌面端论文详情应只保留标题、作者、分类、简洁推荐理由和 Markdown 风格渲染后的摘要。
- 桌面端账户设置应采用轻量弹窗，保存后自动关闭；arXiv 分类、兴趣关键词、排除关键词、关注作者均采用词条式输入。
- arXiv 分类必须从合法 arXiv 分类列表中选择，不能任意填写不存在的分类。
- 待读列表应按账户/Profile 隔离，并支持阅读笔记；PDF 预览功能暂时移除，保留 PDF 下载。
- 关注作者规则：用户可为当前 Profile 设置关注作者；推荐时系统应单独查询关注作者论文，若论文作者命中关注作者，且论文同时匹配当前 Profile 的分类和至少一个兴趣关键词，则额外加入推荐列表，不占用普通推荐 `limit`，也不受普通评分阈值影响。

## 3. 核心目标

### 3.1 当前目标

- 程序化实现每日论文推荐，而不是只生成静态报告。
- 建立清晰的数据获取、推荐排序、数据存储、页面展示和下载流程。
- 提供一个可运行的本地页面，让用户直接查看每日推荐文献。
- 支持快速下载 PDF，或打开 arXiv 原文页面。
- 保留 Markdown 或 JSON 导出能力，作为调试、备份和分享手段。
- 支持本地运行和 GitHub Actions 定时运行。

### 3.2 长期目标

- 支持使用 LLM 生成中文摘要、亮点、贡献点和阅读建议。
- 支持历史推荐浏览、收藏、忽略、已读标记和反馈。
- 支持多 profile、多领域、多用户偏好。
- 支持邮件、飞书、企业微信、Telegram、Notion 等推送方式。
- 支持从用户反馈中调整推荐权重。
- 支持封装成可安装的桌面应用程序。

## 4. 开发要求

### 4.1 功能要求

#### F1. arXiv 数据获取

- 系统必须通过 arXiv 官方 Atom API 获取论文数据。
- 系统必须支持按 arXiv 分类检索，例如 `cs.AI`、`cs.LG`、`cs.CL`。
- 系统必须支持限制单次最大抓取数量。
- 系统必须解析以下字段：
  - arXiv ID
  - 标题
  - 摘要
  - 作者
  - 分类
  - 发布时间
  - 更新时间
  - abs 链接
  - PDF 链接

#### F2. 用户兴趣配置

- 系统必须支持通过配置文件维护用户兴趣。
- 早期配置可以使用 TOML。
- 后续应用页面应支持可视化编辑配置。
- 配置文件至少包含：
  - profile 名称
  - arXiv 分类列表
  - 兴趣关键词
  - 排除关键词
  - 查询时间窗口
  - 最大抓取数量
  - 输出数量
  - 最低推荐分数
- 示例配置文件必须放在 `config.example.toml`。
- 本地真实配置建议使用 `config.toml`，默认不提交到仓库。

#### F3. 推荐与打分

- 系统必须对论文进行可解释打分。
- 当前 MVP 的打分因素包括：
  - arXiv 分类命中加分。
  - 标题关键词命中加分，权重高于摘要。
  - 摘要关键词命中加分。
  - 论文越新，适当加分。
  - 命中排除关键词的论文直接过滤。
- 系统必须保留每篇论文的命中关键词和推荐原因。
- 系统应该避免因为周末、假期或 arXiv 更新间隔导致当日推荐为空，因此时间窗口应优先相对本次抓到的最新论文计算。

#### F4. 数据存储

- 系统必须将抓取结果和推荐结果结构化保存。
- 早期可以使用 JSON 文件作为轻量存储。
- 当功能扩展到历史、收藏、已读和反馈时，可以引入 SQLite。
- 存储层应能支持以下查询：
  - 查看今天推荐。
  - 查看历史某一天推荐。
  - 按关键词、分类、作者搜索。
  - 判断论文是否已推荐过。
  - 记录用户对论文的操作状态。

#### F5. 应用页面

- 系统必须提供一个可交互页面，用于查看每日推荐文献。
- 页面至少包含：
  - 今日推荐列表
  - 标题、作者、分类、日期、分数和命中关键词
  - 摘要展示
  - arXiv 原文链接
  - PDF 下载入口
  - 刷新推荐按钮
  - 历史日期切换
- 页面应支持基础筛选：
  - 按分数排序
  - 按更新时间排序
  - 按分类筛选
  - 按关键词搜索
- 页面应为后续“收藏、忽略、已读、反馈”预留交互空间。

#### F6. 下载能力

- 用户必须可以快速下载论文 PDF。
- 下载行为至少支持两种方式：
  - 直接打开 arXiv PDF 链接。
  - 通过应用下载 PDF 到本地目录。
- 后续可以支持批量下载今日推荐论文。
- 下载目录应可配置。

#### F7. 导出能力

- Markdown 导出保留为过渡和分享能力。
- 系统应支持导出：
  - Markdown 推荐报告
  - JSON 推荐数据
  - 后续可选 CSV
- 导出能力不应成为主产品形态，页面展示才是主要使用入口。

#### F8. 命令行入口

- 系统必须保留命令行入口，方便自动化和调试：

```powershell
python -m auto_arxiv recommend --config config.toml
```

- 后续应新增本地应用启动命令：

```powershell
python -m auto_arxiv serve --config config.toml
```

- 安装后应该支持脚本命令：

```powershell
auto-arxiv recommend --config config.toml
auto-arxiv serve --config config.toml
```

#### F9. 定时运行

- 系统必须支持 GitHub Actions 定时运行。
- 默认运行时间为北京时间每天 08:00。
- GitHub Actions 应支持以下配置来源优先级：
  - `AUTO_ARXIV_CONFIG_TOML` secret
  - 仓库内 `config.toml`
  - `config.example.toml`
- 工作流应保存结构化推荐数据。
- 工作流可以额外生成 Markdown 报告作为 artifact。
- 后续 GitHub Pages 部署时，工作流应能更新页面所需的数据文件。

#### F10. 应用封装

- 核心推荐逻辑必须与界面层解耦。
- Web 页面、本地服务、GitHub 静态部署和桌面应用应尽量复用同一份推荐数据结构。
- 桌面应用封装不得破坏命令行和本地 Web 运行方式。

### 4.2 非功能要求

#### N1. 简单可靠

- 项目早期不引入复杂后端服务。
- 项目早期优先使用文件存储。
- 引入数据库时优先考虑 SQLite。
- 默认依赖应保持克制。

#### N2. 可测试

- 推荐打分、过滤逻辑、配置读取、数据序列化应具备单元测试。
- 页面和 API 成型后，应补充基础端到端测试。
- 每次核心逻辑修改后必须运行：

```powershell
python -m pytest
```

#### N3. 可解释

- 推荐排序不能是黑盒。
- 页面中必须能看出论文为什么被推荐。
- 后续即使接入 LLM，也应保留基础打分、命中关键词和规则解释。

#### N4. 可扩展

- 数据获取、推荐排序、数据存储、接口服务、页面展示、导出功能应拆分模块。
- 新增推送渠道时，不应改动核心推荐逻辑。
- 新增 LLM 摘要时，不应影响无 LLM Key 的基础运行。
- 新增桌面封装时，应复用已有 Web 页面和核心服务。

#### N5. 隐私与配置安全

- `config.toml` 可能包含个人研究兴趣或私有偏好，默认不提交。
- API Key、邮件密码、推送 webhook 等敏感信息必须放入环境变量或 GitHub Secrets。
- GitHub Pages 展示的数据不应包含用户不希望公开的私有配置。
- 如果用户公开部署页面，需要明确区分公开数据和私有配置。

## 5. 建议技术路线

### 5.1 后端与核心逻辑

- Python 版本：`>=3.10`
- 包管理：`pyproject.toml`
- 测试框架：`pytest`
- 本地 Web API 可优先考虑 FastAPI。
- 轻量文件存储优先使用 JSON。
- 历史和用户状态复杂后再引入 SQLite。

### 5.2 前端页面

早期可以采用两种路线之一：

#### 路线 A：轻量静态页面

- Python 生成结构化 JSON。
- 前端使用 HTML、CSS、JavaScript 读取 JSON 并展示。
- 适合 GitHub Pages 部署。
- 适合快速完成可见产品形态。

#### 路线 B：本地 Web 应用

- Python 启动本地服务。
- 页面通过 API 获取推荐数据。
- 适合本地下载、刷新、配置编辑等交互。
- 后续更适合封装桌面应用。

建议先实现路线 A 的数据结构和静态页面，再逐步补充路线 B 的本地服务能力。

### 5.3 桌面封装

当前桌面封装采用 PyInstaller + Python 桌面壳方案：

- PyInstaller 负责生成 Windows exe。
- Python 桌面壳负责启动本地服务、打开页面、刷新推荐和管理配置。
- Web 页面继续负责推荐文献浏览、筛选、搜索和下载。
- 用户数据默认保存在 exe 同目录，包括 `config.toml`、`data/`、`downloads/` 和 `recommendations/`。

后续可以继续评估：

- Tauri：体积较小，但需要 Rust 工具链。
- Electron：生态成熟，但体积较大。
- 内嵌浏览器控件：让用户体验更接近完整桌面软件。

说明：

- `export.py`：负责 Markdown、JSON、CSV 等导出。

## 7. 推荐流程

最终流程应从“生成报告”升级为“生成可被程序消费的数据，并在页面展示”：

```text
读取配置
  ↓
请求 arXiv API
  ↓
解析论文数据
  ↓
结构化保存原始论文
  ↓
按时间窗口和排除词过滤
  ↓
计算推荐分数和推荐原因
  ↓
保存每日推荐数据
  ↓
页面读取推荐数据
  ↓
用户查看、筛选、搜索、下载
  ↓
可选导出 Markdown / JSON / CSV
```

## 8. 配置规范

示例：

```toml
[profile]
name = "My Research Radar"
categories = ["cs.AI", "cs.LG", "cs.CL"]
keywords = [
  "large language model",
  "agent",
  "retrieval augmented generation",
  "alignment",
  "reasoning",
]
exclude_keywords = [
  "position paper",
]

[search]
days_back = 2
max_results = 100

[output]
limit = 10
min_score = 2.0
directory = "recommendations"
data_directory = "data"
download_directory = "downloads"
```

字段说明：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `profile.name` | string | 当前推荐 profile 名称 |
| `profile.categories` | string[] | arXiv 分类 |
| `profile.keywords` | string[] | 兴趣关键词 |
| `profile.exclude_keywords` | string[] | 排除关键词 |
| `search.days_back` | int | 推荐时间窗口 |
| `search.max_results` | int | 单次最多抓取数量 |
| `output.limit` | int | 每日最多推荐论文数量 |
| `output.min_score` | float | 最低推荐分数 |
| `output.directory` | string | Markdown 导出目录 |
| `output.data_directory` | string | 结构化数据输出目录 |
| `output.download_directory` | string | PDF 下载目录 |

## 9. 版本规划

### v0.1：推荐链路验证版

- arXiv 抓取。
- 本地 TOML 配置。
- 关键词打分。
- Markdown 日报导出。
- GitHub Actions 定时运行。
- 单元测试覆盖核心打分逻辑。

### v0.2：结构化数据版

- 输出每日推荐 JSON。
- 保存论文历史数据。
- 增加论文去重。
- 为页面展示定义稳定数据结构。

### v0.3：静态页面版

- 增加 `public/` 静态页面。
- 页面展示今日推荐和历史推荐。
- 支持搜索、排序、分类筛选。
- 支持打开 PDF 和 arXiv 页面。

### v0.4：本地 Web 应用版

- 增加本地服务启动命令。

### v0.5：阅读体验增强版

- 增加中文摘要。
- 增加“为什么推荐”说明。

- 支持收藏、忽略、已读标记。

### v0.6：桌面应用封装版

- 支持邮件、飞书或企业微信推送。

## 10. 当前不做的事情

- 暂不做用户登录系统。
- 暂不做复杂云端多用户平台。
- 暂不实现复杂机器学习推荐模型。
- 暂不强依赖任何 LLM 服务。
- 暂不做复杂安装器和自动更新。
- 暂不抓取非 arXiv 来源，除非后续明确需要。

注意：Web 页面是当前目标的一部分，不再属于“暂不做”。

## 11. 验收标准

### 11.1 当前 MVP 验收

- 可以通过示例配置生成推荐结果。
- 可以导出 Markdown 报告。
- 没有论文命中时，也能生成结构完整的空结果。
- `python -m pytest` 全部通过。
- README 能指导用户完成本地运行。
- GitHub Actions 能在无人工介入的情况下定时生成推荐数据或报告。

### 11.2 程序化应用验收

- 用户可以打开本地或 GitHub 部署页面查看每日推荐文献。
- 页面可以展示标题、作者、摘要、分类、分数、命中关键词和推荐原因。
- 用户可以快速打开或下载论文 PDF。
- 用户可以查看历史推荐。
- 页面数据来自结构化文件或本地 API，而不是手写 Markdown。
- Markdown 只作为导出选项，而不是唯一入口。

## 12. 后续开发优先级

下一步建议按以下顺序推进：

1. 修正和完善 README 的中文显示与项目目标描述。
2. 增加结构化 JSON 输出，作为页面数据源。
3. 增加历史去重，避免重复推荐同一篇论文。
4. 增加 `public/` 静态页面，展示每日推荐。
5. 增加 GitHub Pages 部署流程。
6. 增加本地 Web 服务和 PDF 下载能力。
7. 增加 LLM 摘要模块，但保持可选。
8. 评估内嵌浏览器、安装器和自动更新方案。
